# platformio-mcp 后续建设原则与路线判断

## 1. 项目目标与用户画像

`platformio-mcp` 的目标不是做一个普通的 PlatformIO CLI 包装器，而是逐步建设成一个面向 AI Agent 的嵌入式软硬联调执行层。

未来期望这个项目支持用户通过 Agent 尽量完成：

- 项目分析
- 配置理解
- 代码修改
- 编译
- 烧录
- 监控串口输出
- 发现问题
- 验证结果
- 重试修复

项目面向的不是愿意手工处理大量硬件细节的嵌入式专家，而是：

- 不想深度处理硬件细节的开发者
- 希望 Agent 尽量替代手工联调摩擦的用户
- 更关注“能不能把事情做成”，而不是“底层链路是否都亲自掌握”的使用者

因此，这个项目必须优先追求：

- 低硬件门槛
- 真实可用
- 结构化反馈
- 常见场景优先
- Agent 可直接消费的结果，而不是原始 CLI 文本

## 2. 项目分层定位

这个项目应按三层理解，而不是按单一工具理解。

### 第一层：Execution Layer

当前主建设层。

负责提供：

- inspect project
- list environments
- doctor
- build
- clean
- upload
- monitor
- library 管理
- device / port 识别
- 结构化结果返回

### 第二层：Workflow Layer

后续逐步建设。

负责提供：

- 调试 session
- 运行结果验证
- 自动重试
- 错误归因
- `build -> upload -> monitor -> verify` 闭环编排
- 面向用户目标的任务拆解

### 第三层：Visual / Collaboration Layer

更后续建设。

负责提供：

- VS Code 扩展
- 项目状态面板
- 日志面板
- 设备状态与操作界面
- Agent 操作过程可视化
- 高风险动作确认

当前阶段主要建设第一层，并为第二层与第三层预留结构。

## 3. 当前阶段判断

当前项目处于 `Execution Layer / P0 中段`。

判断依据：

- 非硬件依赖链路已经比较完整，项目探测、环境解析、编译、库管理、设备枚举、结构化响应已有基础
- Windows + PlatformIO + VS Code 常见安装场景已开始兼容
- 非硬件依赖的真实 PlatformIO smoke test 已经基本打通
- 但真实开发板闭环尚未完成，`upload -> monitor -> 读取业务串口输出` 仍然是关键门槛

这意味着：

- 项目还不能被视为成熟的软硬联调 Agent 执行层
- 也还不适合过早向高阶自治调试平台扩张
- 近期最重要的工作不是增加工具数量，而是把常见板卡路径的真实闭环做扎实

## 4. 核心建设原则

### 4.1 真实可用优先

优先做：

- 能 build
- 能 upload
- 能 monitor
- 能明确失败原因
- 能验证结果

而不是优先堆很多概念性工具名或高阶工作流概念。

### 4.2 常见场景优先

优先支持：

- Arduino
- ESP32
- RP2040
- 常见 PlatformIO 项目
- 串口日志
- 基础 upload
- 基础运行验证

不以 JTAG / OpenOCD / 复杂调试器链路作为主路径。

### 4.3 Agent 可执行优先

所有底层能力都应面向 Agent 的调用与决策，而不是只面向人工命令行使用。

这意味着结果必须尽量结构化，并能够表达：

- 发生了什么
- 成功还是失败
- 失败在哪
- 下一步建议是什么

### 4.4 用户硬件负担最小化

用户不应承担过多以下复杂性：

- 端口猜测
- environment 猜测
- 波特率猜测
- upload 链路经验判断
- 是否需要进入 boot 模式的隐式知识

这些复杂性应尽量由执行层内部吸收，并转化为自动判断、明确提示和结构化状态。

### 4.5 严禁伪造结果

没有真实硬件，就明确说没有。

没有真实验证成功，就明确说没有。

不能把“理论可行”写成“已经完成”。

### 4.6 长期演进友好

当前每一轮建设都要兼顾后续可演进性：

- 为 Workflow Layer 提供稳定底层能力
- 为 VS Code 扩展提供可消费状态接口
- 为高层 Agent 提供可验证、可重试、可归因的数据基础

## 5. 优先能力方向

### P0：把 Execution Layer 做扎实

优先建设：

- 真实 build / upload / monitor 闭环
- 更稳的端口识别
- 更清晰的 doctor 与项目探测
- 更好的错误摘要
- 结构化返回稳定化
- Windows 常见环境兼容性

这是当前最重要的主线。

### P1：为 Workflow Layer 做准备

在 P0 扎实后推进：

- 调试 session
- 验证规则基础能力
- 自动重试支撑接口
- 错误归因基础能力
- 代码修改后重新 build / upload / verify 的底层支撑

### P2：为 VS Code 扩展做准备

在核心执行层和状态模型稳定后推进：

- 项目摘要接口
- 设备摘要接口
- 最近任务状态接口
- 可供 UI 直接展示的结构化结果
- 高风险动作确认边界

### P3：更高阶联调能力

更后续再推进：

- 高阶 debug workflow
- 自动验证 / 自动修复 / 自动重试
- 更强的用户目标驱动执行
- 更复杂的运行时设备交互协议

## 6. 暂缓方向

当前阶段明确暂缓以下方向：

### 高阶自治调试 Agent

原因：

- 底层真实硬件闭环还未完全稳定
- 错误语义和验证结果模型还不够成熟

### 复杂调试器链路

例如：

- JTAG
- OpenOCD
- GDB
- 板载复杂调试器集成

原因：

- 不符合当前“低硬件门槛”的主路径
- 对常见用户场景收益不如串口闭环高

### 重型 VS Code UI

原因：

- 现在更需要稳定状态接口，而不是先设计复杂界面

### 复杂运行时设备控制协议

原因：

- 当前主问题仍是 `build/upload/monitor/verify` 闭环，而不是运行时控制抽象

### 大而全工作流编排器

原因：

- 底层执行层和真实验证结果还不足以支撑稳定的上层编排

## 7. 与 VS Code 扩展和高层 Agent 的关系

### 对 VS Code 扩展的关系

当前阶段不做重型 UI，但必须开始沉淀未来扩展可直接消费的接口。

应优先预留：

- 项目摘要
- 设备摘要
- 最近一次 build / upload / monitor / verify 结果
- 明确的失败类别
- 明确的下一步建议

未来扩展不应重新解析原始 CLI 输出，而应尽量直接消费执行层结构化结果。

### 对高层 Agent 的关系

未来高层 Agent 的成功依赖于当前层提供：

- 真实可执行的原子能力
- 真实可验证的结果
- 统一的状态字段
- 明确的失败语义
- 可重试的动作边界

如果当前层只是 CLI 薄包装，后续高层 Agent 将不可避免地重新拼接命令知识、重写错误判断逻辑，并把复杂性重新暴露给用户。

## 8. 后续推进要求

后续每轮推进应遵循以下要求：

### 先规划，再实施

除非任务极小，否则先明确：

- 目标
- 范围
- 依赖
- 风险
- 验证方式

### 每轮只解决清晰的问题

不要在一轮里同时做：

- 大重构
- 新功能泛滥
- 文档大范围改写
- UI 设计发散

除非明确要求。

### 每轮都必须可验证

每轮结束后都要明确：

- 改了什么
- 怎么验证
- 哪些通过
- 哪些没法验证
- 剩余风险是什么

### 始终以“Execution Layer 做厚，再向 Workflow Layer 抬升”为主线

这是当前阶段最重要的工程路线约束。

在真实硬件闭环未完成前，不应把重心切到高阶自治或重型可视化层。
